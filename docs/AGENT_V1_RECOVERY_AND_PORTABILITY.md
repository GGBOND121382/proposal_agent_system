# 智能体 V1 候选：恢复与跨模型可迁移性修复

## 目标

本轮以智能体为交付物。申请书仅作为运行产物。系统必须支持：

1. 使用任意 OpenAI-compatible 模型 API；
2. 新课题冷启动时不读取旧正文或旧模型响应；
3. 安全检查、进程中断或机器重启后，从已提交模型 Run 精确续跑；
4. 多个 WF-4 重跑共存时，WF-5、导出和验收始终消费同一冻结候选集合。

## 两种模式

- `FRESH_GENERATION`：跨模型、跨课题冷启动验收。发现相同 call key 的旧数据库提交或响应证据时直接阻断，不静默复用。
- `RESUME_FROM_CHECKPOINT`：生产恢复。只允许复用完整请求身份一致的已提交结果。

请求身份包括项目、工作流、Prompt ID、Prompt 版本、输入哈希、配置模型 ID、配置端点 ID和供应商模型名。

## 上一轮发现并固化的通用修复

- Schema 脚手架值与真实空数组区分；
- Producer/Critic 对象层级映射；
- 五组并发使用 `ContextVar` 隔离工作流和章节，禁止跨组串稿；
- Integration Critic 从父工作流绑定的子工作流汇总14章候选；
- 章节候选 ID、Polish Run 和 Expression Critic Run 一一绑定；
- WF-5 显式绑定一个完成且通过全文审查的 WF-4；
- 导出器不再按“项目全局最新章节”混合多轮候选；
- 后验收、质量阻断和人工 Gate 按冻结工作流隔离；
- PDF 表格单元格采用局部有序重构验证，避免文本抽取横向交错造成误报；
- 旧版单工作流检查点可从精确 Run ID 迁移，标记为 `MIGRATED_LEGACY_CHECKPOINT`；
- 模型响应、检查点复用和迁移均写入来源、哈希与审计事件。

## 正文复用边界

安全中断后复用智能体此前已生成、已通过 Schema 校验并已提交数据库的章节，是正常恢复能力。生产路径禁止读取无来源约束的外部预写正文文件。
