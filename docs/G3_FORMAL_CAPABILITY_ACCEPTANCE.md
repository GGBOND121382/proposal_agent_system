# G3 正式能力验收

## 1. 验收范围

G3 是完整申请书能力的最终证明，不复用固定模型响应，也不以编排模拟、测试样例或人工改写正文代替模型能力。正式运行必须在一组完整、公开、可重复的基准材料上依次完成项目定义、事实提取、实时公开研究、论证架构、完整章节规划、五组并发写作、分章审查和定向修订、每三章跨章审查、全文审查、最终复核以及 DOCX/PDF 导出后验收。

## 2. 正式运行条件

`scripts/run_g3_formal_acceptance.py` 只接受以下环境：

- `MODEL_RUNTIME_MODE=LIVE`；
- `CAPABILITY_ACCEPTANCE_MODE=true`；
- 可用的两个 OpenAI-compatible 模型路由；
- 实时公开检索提供方为 Crossref 或 SearXNG；
- LibreOffice、Poppler 和 Chromium 可用；
- 所有输入材料均标记为 PUBLIC，并有文件哈希。

预检只记录配置项是否可用，不记录认证内容。任何条件不满足时只能生成 `BLOCKED_CONFIGURATION`，不得产生 G3 PASS。

## 3. 公开基准材料

`scripts/build_g3_public_materials.py` 生成一个民用动态城市配送研究基准，包含申报规则、项目定义、方法与实验、公开基础、研究要求、14 章节空白申请书和参考结构。材料不包含个人信息、内部项目数据或未公开成果。

实时公开研究从学术元数据服务获取当前记录，归档原始结果、标准化元数据、来源哈希、查询覆盖和 Claim 绑定。固定记录文件和测试连接器不能进入正式结果。

## 4. 每三章审查

正式 WF-4 将冻结的 14 个章节按顺序划分为 `3+3+3+3+2` 五批。每批使用独立 Integration Critic Run 和独立候选集合哈希。章节发生修订后，包含该章节的批次记录立即失效，必须重新审查。五批全部 PASS 后才允许执行全文 Integration Critic。

## 5. 模型调用证据

每次模型调用保存：

- 完整质量上下文哈希；
- 实际模型输入及其哈希；
- 输入压缩规则和压缩前后长度；
- 完整输出 Schema；
- 原始响应文本及其哈希；
- 解析对象及其哈希；
- 模型、端点、状态、耗时和调用键。

针对上下文长度约束，只允许删除重复的可选说明并压缩长引文；完整上下文仍用于确定性质量检查并写入 Trace。模拟模式不应用正式输入压缩。

## 6. PASS 条件

机器可读报告 `G3_FORMAL_CAPABILITY_ACCEPTANCE.json` 只有在以下条件全部成立时为 PASS：

1. WF-1 至 WF-5 全部完成；
2. 所有 Prompt Run 均来自 LIVE 模型，且无错误或阻断状态；
3. 原始请求和响应证据完整；
4. 完整公开材料均已导入；
5. 实时公开研究归档、哈希和覆盖检查通过；
6. 14 章节和 5 个编制组完成；
7. 5 个跨章批次全部通过；
8. 全文 Integration Critic 通过；
9. 所有 P0/P1 均已关闭；
10. DOCX、PDF、交付包和逐页视觉证据通过；
11. SQLite、Trace、环境清单、材料清单和恢复包可以复核。

代码和普通测试通过只能证明 G3 运行器可用，不能替代上述真实运行结果。
